这是一个实现容器应用启动优化的项目。（目前仅面向python应用）

场景：
在卫星计算的领域，应用常常通过容器形式部署在卫星搭载的计算载荷（通常是树莓派或jetson nano）上，但受限于在轨能量供给与功耗预算，计算载荷往往无法持续上电运行，工程上通常采用按需或基于任务窗口的受控供电策略。容器镜像和应用都是离线存储在计算载荷本地的，不需要网络拉取。

问题：
窗口式供电意味着计算载荷频繁经历“断电—上电”的周期。每次上电后，系统往往处于近似“冷启动/无缓存”的初始状态，导致容器应用需要重复完成较长的启动路径，从而显著压缩任务窗口内真正用于计算的有效时间。具体表现为两类瓶颈：

1. 存在频繁冷启动：断电会清空内存态与页缓存，上电后需重复执行 OS/服务初始化、容器运行时初始化、镜像层解析与挂载、依赖加载等步骤。
2. 容器应用启动开销大：星载容器化应用普遍存在
   - 依赖体积大：通常引入较多运行库/模型（例如 ML 推理依赖、图像处理库、脚本运行时等），容器启动时依赖加载时间过长


为解决上述两个问题，我们提出一种面向在轨应用执行窗口供电的快速恢复型容器运行环境框架，分别优化“上电前准备（构建/部署）”与“上电后执行（运行）”两个阶段，使应用能够从“冷启动”尽可能转向“warm start”，最大化任务窗口内的有效计算时间。



方法：
- 构建阶段：生成 *镜像文件
  - 检查点自动发现：
  镜像注入wrapper程序，替换容器入口点
  wrapper程序负责导入依赖，判断checkpoint-ready状态，并将启动参数传递给真正入口程序；
  - 镜像交付物 设计与布局优化：设计可预读的镜像格式，实现系统启动时快速填充page cache
    - 分析checkpoint的restore过程，构建启动热点文件列表；
    - 将 rootfs + checkpoint 产物封装为只读底层对象（erofs），同时重排镜像文件分布，将热点文件顺序靠前排列；
- 执行期：最小化 [上电, 可用] 时间
  - fast cache warm: 
  上电早期运行preheater，顺序读镜像文件热点文件（如前n字节）；该阶段尽量与系统/运行时启动并行；
  - restore: 
  跳过依赖初始化阶段，并从warmed cache环境恢复容器；
